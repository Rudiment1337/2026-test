# Лекция 12: Тестирование и логгирование
1. JUnit 5 - Архитектура и концепции
**Теоретическая основа:**
`JUnit 5` представляет собой полную переработку фреймворка тестирования с модульной архитектурой. Ключевые философские аспекты:

**Архитектурные компоненты:**

- `JUnit Platform` - обеспечивает инфраструктуру для запуска тестов на JVM
- `JUnit Jupiter` - новая модель программирования и механизм расширений
- `JUnit Vintage` - обеспечивает обратную совместимость с JUnit 3 и 4

- JUnit Jupiter: включает новые модели программирования и расширения для написания тестов. В нем есть все новые аннотации junit и TestEngine реализация для запуска тестов, написанных с этими аннотациями.

- Платформа JUnit: чтобы иметь возможность запускать тесты junit, IDE, инструменты сборки или плагины должны включать и расширять API платформы. Он определяет TestEngine API для разработки новых фреймворков тестирования, работающих на платформе. Он также предоставляет средство запуска консоли для запуска платформы из командной строки и создания подключаемых модулей для Gradle и Maven.

- JUnit Vintage: его основная цель - поддерживать выполнение на платформе JUnit 5 тестов, написанных для JUnit 3 и JUnit 4. Это есть обратная совместимость.

**Установка**
Вы можете использовать JUnit 5 в своем проекте Maven или Gradle, включив как минимум следующие зависимости:

- junit-jupiter-api: это основной модуль, в котором расположены все основные аннотации, такие как @Test, аннотации и утверждения метода жизненного цикла.

- junit-jupiter-engine: он имеет реализацию тестового движка, которая требуется во время выполнения для выполнения тестов.

- junit-platform-suite: поддержка @Suite, предоставляемая этим модулем, чтобы сделать средство запуска JUnitPlatform устаревшим.

**Аннотации**
JUnit 5 предлагает следующие аннотации для написания тестов.

Аннотации|Описание
---|---
@BeforeEach|Аннотированный метод будет запускаться перед каждым тестовым методом в тестовом классе.
@AfterEach|Аннотированный метод будет запускаться после каждого тестового метода в тестовом классе.
@BeforeAll|Аннотированный метод будет запущен перед всеми тестовыми методами в тестовом классе. Этот метод должен быть статическим.
@AfterAll|Аннотированный метод будет запущен после всех тестовых методов в тестовом классе. Этот метод должен быть статическим.
@Test|Он используется, чтобы пометить метод как тест junit.
@DisplayName|Используется для предоставления любого настраиваемого отображаемого имени для тестового класса или тестового метода
@Disable|Он используется для отключения или игнорирования тестового класса или тестового метода из набора тестов.
@Nested|Используется для создания вложенных тестовых классов
@Tag|Пометьте методы тестирования или классы тестов тегами для обнаружения и фильтрации тестов.
@TestFactory|Отметить метод - это тестовая фабрика для динамических тестов.

**Написание тестов**
``` java
public class AppTest {

    @BeforeAll
    static void setup(){
        System.out.println("@BeforeAll executed");
    }

    @BeforeEach
    void setupThis(){
        System.out.println("@BeforeEach executed");
    }

    @Tag("DEV")
    @Test
    void testCalcOne()
    {
        System.out.println("======TEST ONE EXECUTED=======");
        Assertions.assertEquals( 4 , Calculator.add(2, 2));
    }

    @Tag("PROD")
    @Disabled
    @Test
    void testCalcTwo()
    {
        System.out.println("======TEST TWO EXECUTED=======");
        Assertions.assertEquals( 6 , Calculator.add(2, 4));
    }

    @AfterEach
    void tearThis(){
        System.out.println("@AfterEach executed");
    }

    @AfterAll
    static void tear(){
        System.out.println("@AfterAll executed");
    }
}
```

**Принципы тестирования в JUnit 5:**

Тест как гражданин первого класса - методы тестирования могут возвращать значения и объявлять исключения

Расширяемость через механизм расширений - заменяет устаревшие @Rules

Условное выполнение тестов - возможность пропускать тесты на основе условий

Группировка и фильтрация - теги и отображение для организации тестов

Модель жизненного цикла тестов:
- `TestInstance lifecycle` - фундаментальное изменение в управлении состоянием:
- `PER_METHOD (по умолчанию)` - новый экземпляр тестового класса для каждого метода
- `PER_CLASS` - один экземпляр для всех методов тестового класса

Диспетчеризация тестов - JUnit 5 использует более интеллектуальный механизм обнаружения и выполнения тестов с поддержкой динамических тестов.

**Assertions**

`Assertions` (утверждения) позволяют сравнить ожидаемый результат с фактическим результатом теста.

Для того чтобы держать вещи простыми, все утверждения JUnit Jupiter являются static методы в класса org.junit.jupiter.Assertions, например `assertEquals()`, `assertNotEquals()`.
``` java
void testCase()
{
    //Test will pass
    Assertions.assertNotEquals(3, Calculator.add(2, 2));

    //Test will fail
    Assertions.assertNotEquals(4, Calculator.add(2, 2), "Calculator.add(2, 2) test failed");

    //Test will fail
    Supplier<String> messageSupplier  = () -> "Calculator.add(2, 2) test failed";
    Assertions.assertNotEquals(4, Calculator.add(2, 2), messageSupplier);
}
```

**Assumptions**

Класс `Assumptions` (предположения) предоставляет static методы для поддержки выполнения условного теста на основе предположений. Неуспешное предположение приводит к прерыванию теста.

Предположения обычно используются всякий раз, когда нет смысла продолжать выполнение данного метода тестирования. В отчете о тестировании эти тесты будут отмечены как пройденные.

Предположения класс имеет три таких методов: `assumeFalse()`, `assumeTrue()` и `assumingThat()`
``` java
public class AppTest {
    @Test
    void testOnDev()
    {
        System.setProperty("ENV", "DEV");
        Assumptions.assumeTrue("DEV".equals(System.getProperty("ENV")), AppTest::message);
    }

    @Test
    void testOnProd()
    {
        System.setProperty("ENV", "PROD");
        Assumptions.assumeFalse("DEV".equals(System.getProperty("ENV")));
    }

    private static String message () {
        return "TEST Execution Failed :: ";
    }
}
```

1. Mockito - Теория мокинг-фреймворков
**Концептуальная модель:**
Mockito реализует паттерн "Test Double", который включает несколько типов объектов-заменителей:

Классификация Test Doubles:

- Dummy - объекты-пустышки, никогда не используются

- Fake - работающая реализация с упрощенной логикой

- Stub - предоставляет предопределенные ответы на вызовы

- Mock - объекты с запрограммированными ожиданиями и верификацией

**Основы Mockito**
Для начала, создадим простой мок-объект. Представим, у нас есть интерфейс DataService, который мы хотим замокировать:

``` java
public interface DataService {
    double retrieveData();
}
```
Используя Mockito, мы можем создать мок этого интерфейса очень просто:

``` java
DataService mockDataService = Mockito.mock(DataService.class);
```
`mockDataService` - это мок-объект, который можно использовать в тестах. Он будет иметь поведение по умолчанию для всех методов (например, возвращать 0 для методов, возвращающих double).

Чтобы мок был полезен, нам нужно определить его поведение. Допустим, мы хотим, чтобы метод `retrieveData()` возвращал определенное значение. Сделать это можно следующим образом:

``` java
Mockito.when(mockDataService.retrieveData()).thenReturn(15.0);
```
Теперь при вызове `retrieveData()` на мок-объекте mockDataService, он вернет 15.0.

Теперь, когда у нас есть мок и мы настроили его поведение, мы можем использовать его в нашем тесте. Допустим, у нас есть класс `DataProcessor`, который зависит от `DataService`. Мы можем использовать мок `DataService` для тестирования `DataProcessor`:
``` java
public class DataProcessor {
    private DataService dataService;

    public DataProcessor(DataService dataService) {
        this.dataService = dataService;
    }

    public double processData() {
        return dataService.retrieveData() * 2;
    }
}
```
В тесте:
``` java
@Test
public void testProcessData() {
    DataService mockDataService = Mockito.mock(DataService.class);
    Mockito.when(mockDataService.retrieveData()).thenReturn(15.0);

    DataProcessor processor = new DataProcessor(mockDataService);
    double result = processor.processData();

    assertEquals(30.0, result, 0.01);
}
```
Проверяем, что метод `processData()` корректно удваивает значение, полученное от `DataService`.

**Методы стаббинга в Mockito**
Метод `when-thenReturn` - самая часто используемая форма стаббинга в Mockito. Он позволяет определить возвращаемое значение для метода мок-объекта.

Допустим, у нас есть интерфейс `CalculatorService`:
``` java
public interface CalculatorService {
    double add(double input1, double input2);
}
```
Мы можем создать мок для этого интерфейса и определить поведение метода add таким образом:
``` java
@Test
public void whenThenReturnExample() {
    CalculatorService calculatorService = Mockito.mock(CalculatorService.class);

    // Стаббинг: определение поведения
    Mockito.when(calculatorService.add(10.0, 20.0)).thenReturn(30.0);

    // Проверка: метод add возвращает 30.0
    assertEquals(30.0, calculatorService.add(10.0, 20.0), 0.01);
}
```
Здесь мы задаем, что при вызове `add(10.0, 20.0)` мок должен вернуть `30.0`.

В то время как `when-thenReturn` идеально подходит для большинства сценариев, иногда мы сталкиваемся с некоторыми ограничительными нюансами, особенно когда работаем с методами, которые возвращают `void`. В этих случаях полезен `doReturn-when`.

Предположим, у нас есть метод `void printSum(double input1, double input2)`, который не возвращает значение, но мы хотим удостовериться, что он был вызван с определенными параметрами. Мы можем использовать `doReturn-when` для этой цели:
``` java
@Test
public void doReturnWhenExample() {
    CalculatorService calculatorService = Mockito.mock(CalculatorService.class);

    // Стаббинг с использованием doReturn-when
    Mockito.doNothing().when(calculatorService).printSum(10.0, 20.0);

    // Вызов метода
    calculatorService.printSum(10.0, 20.0);

    // Верификация: метод был вызван с заданными параметрами
    Mockito.verify(calculatorService).printSum(10.0, 20.0);
  }
  ```
Используем `doNothing()`, чтобы указать, что при вызове `printSum(10.0, 20.0)` ничего не должно происходить, но мы можем проверить, что вызов действительно был сделан.

**Верификация вызовов методов в Mockito**
Допустим, у нас есть интерфейс `OrderService`, который мы хотим протестировать:

``` java
public interface OrderService {
    void placeOrder(Order order);
    int getOrderCount();
}
```
Когда мы пишем тест, нам может понадобиться убедиться, что метод `placeOrder` был действительно вызван. Вот как мы можем это сделать с помощью Mockito:
``` java
@Test
public void verifyMethodCall() {
    // Создание мока
    OrderService orderService = Mockito.mock(OrderService.class);

    // Создание тестового заказа
    Order testOrder = new Order("TestItem", 3);

    // Имитация вызова метода
    orderService.placeOrder(testOrder);

    // Верификация: был ли вызван метод placeOrder с testOrder
    Mockito.verify(orderService).placeOrder(testOrder);
}
```
Здесь мы не только создали мок `OrderService`, но и проверили, что метод `placeOrder` был вызван с определенным заказом.

Mockito позволяет нам идти еще дальше: мы можем проверить количество вызовов метода и точные параметры, с которыми он был вызван. Это полезно, когда один и тот же метод вызывается несколько раз с разными параметрами.

``` java
@Test
public void verifyNumberOfMethodCalls() {
    OrderService orderService = Mockito.mock(OrderService.class);

    // Имитация вызовов
    orderService.placeOrder(new Order("Item1", 1));
    orderService.placeOrder(new Order("Item2", 2));

    // Верификация: метод placeOrder был вызван дважды
    Mockito.verify(orderService, Mockito.times(2)).placeOrder(Mockito.any(Order.class));

    // Верификация: метод getOrderCount ни разу не был вызван
    Mockito.verify(orderService, Mockito.never()).getOrderCount();
}
```
В этом примере мы проверяем, что `placeOrder` был вызван ровно два раза с любыми заказами и что `getOrderCount` не был вызван вообще.

**Мокирование исключений**
Мокирование исключений полезно, когда вы хотите тестировать поведение вашего кода в условиях возникновения исключения.

Допустим, у нас есть интерфейс `FileReader`, который читает файл и может выбрасывать `IOException`:
``` java
public interface FileReader {
    String readFile(String path) throws IOException;
}
```
Теперь, представим, что мы хотим протестировать поведение нашего класса `FileProcessor`, который зависит от `FileReader`, в ситуации, когда `IOException` выбрасывается.
``` java
public class FileProcessor {
    private FileReader fileReader;

    public FileProcessor(FileReader fileReader) {
        this.fileReader = fileReader;
    }

    public String processFile(String path) {
        try {
            return "Processed: " + fileReader.readFile(path);
        } catch (IOException e) {
            return "Error";
        }
    }
}
```
Чтобы протестировать этот случай, мы можем использовать `Mockito` для мокирования `IOException`:
``` java
@Test
public void whenIOException_thenReturnsError() throws IOException {
    // Создание мока
    FileReader mockFileReader = Mockito.mock(FileReader.class);

    // Мокирование исключения
    Mockito.when(mockFileReader.readFile(Mockito.anyString())).thenThrow(new IOException());

    // Тестирование
    FileProcessor fileProcessor = new FileProcessor(mockFileReader);
    String result = fileProcessor.processFile("test.txt");

    // Проверка
    assertEquals("Error", result);
}
```
В этом тесте мы гарантируем, что когда метод `readFile` вызывается, `IOException` выбрасывается, и `FileProcessor` корректно обрабатывает это исключение.

Mockito также позволяет мокировать исключения для конкретных параметров вызываемого метода. Это полезно, когда поведение метода зависит от предоставленных аргументов.
``` java
@Test
public void whenSpecificArgument_thenThrowException() throws IOException {
    // Создание мока
    FileReader mockFileReader = Mockito.mock(FileReader.class);

    // Мокирование исключения для конкретного аргумента
    Mockito.when(mockFileReader.readFile("specific.txt")).thenThrow(new IOException());

    // Тестирование
    FileProcessor fileProcessor = new FileProcessor(mockFileReader);
    String result = fileProcessor.processFile("specific.txt");

    // Проверка
    assertEquals("Error", result);
}
```
Здесь мы гарантируем, что `IOException` будет выброшен только тогда, когда `readFile` вызывается с `"specific.txt"` в качестве аргумента.

**ArgumentMatchers в Mockito**
**ArgumentMatchers** - это специальные условия, которые мы можем применять к аргументам вызываемых методов. Они позволяют определить правила для любых аргументов или их определенных комбинаций.

Интерфейс `UserService`, который мы хотим протестировать:
``` java
public interface UserService {
    User findUser(String username, int age);
}
```
Теперь, мы хотим замокировать этот сервис и определить поведение метода `findUser`.

Допустим, нам не важно, с какими аргументами вызывается метод - мы хотим вернуть определенного пользователя для любых значений.
``` java
@Test
public void whenUsingAnyMatcher_thenUserIsReturned() {
    UserService mockUserService = Mockito.mock(UserService.class);
    User expectedUser = new User("Alice", 30);

    // Стаббинг с использованием any()
    Mockito.when(mockUserService.findUser(Mockito.anyString(), Mockito.anyInt())).thenReturn(expectedUser);

    // Вызов метода с любыми аргументами
    User result = mockUserService.findUser("Bob", 25);

    // Проверка
    assertEquals(expectedUser, result);
}
```
Вне зависимости от передаваемых в `findUser` аргументов, всегда будет возвращаться expectedUser.

Теперь, если нам нужно определить поведение метода для конкретных аргументов:
``` java
@Test
public void whenUsingEqMatcher_thenSpecificUserIsReturned() {
    UserService mockUserService = Mockito.mock(UserService.class);
    User specificUser = new User("Bob", 25);

    // Стаббинг: метод должен быть вызван с конкретными аргументами
    Mockito.when(mockUserService.findUser(Mockito.eq("Bob"), Mockito.eq(25))).thenReturn(specificUser);

    // Вызов метода с конкретными аргументами
    User result = mockUserService.findUser("Bob", 25);

    // Проверка
    assertEquals(specificUser, result);
}
```
Здесь `findUser` возвращает `specificUser` только если вызывается с аргументами `"Bob"` и `25`.

ArgumentMatchers также полезны при верификации вызовов методов:
``` java
@Test
public void verifyWithArgumentMatchers() {
    // Создание мока
    UserService mockUserService = Mockito.mock(UserService.class);

    // Вызов метода
    mockUserService.findUser("Alice", 30);

    // Верификация: был ли метод вызван с определенными аргументами
    Mockito.verify(mockUserService).findUser(Mockito.eq("Alice"), Mockito.eq(30));
}
```
**Спай-объекты (Spying) в Mockito**
**Спай-объекты** - это специальные моки, которые в основном ведут себя как обычные объекты, но позволяют переопределять поведение отдельных методов. Они мегаполезны, когда вам нужно тестировать реальный объект, но при этом изменить поведение некоторых его частей.

Допустим, у нас есть класс `ListManager`, который мы хотим протестировать:
``` java
public class ListManager {
    public List<String> createList() {
        return new ArrayList<>();
    }

    public int getListSize(List<String> list) {
        return list.size();
    }
}
```
Мы можем создать спай для этого класса следующим образом:
``` java
@Test
public void spyExample() {
    ListManager spyListManager = Mockito.spy(new ListManager());

    // Использование реального метода
    List<String> list = spyListManager.createList();
    list.add("Item");

    // Переопределение поведения метода getListSize
    Mockito.when(spyListManager.getListSize(list)).thenReturn(100);

    // Тестирование
    int size = spyListManager.getListSize(list);

    // Проверка
    assertEquals(100, size);
}
```
Мы создали спай `ListManager`, который использует реальный метод `createList`. Однако, мы переопределили поведение метода `getListSize`, чтобы он возвращал `100`, независимо от реального размера списка.

Также как и с моками, мы можем использовать верификацию с спай-объектами:
``` java
@Test
public void verifySpy() {
    // Создание спай-объекта
    ListManager spyListManager = Mockito.spy(new ListManager());

    // Вызов метода
    List<String> list = spyListManager.createList();
    list.add("Item");
    spyListManager.getListSize(list);

    // Верификация вызова метода
    Mockito.verify(spyListManager).getListSize(list);
}
```
Здесь мы проверяем, что метод `getListSize` был действительно вызван на спай-объекте.

**Тестирование с использованием аннотаций**
Аннотация @Mock используется для создания мок-объектов. Вместо явного вызова Mockito.mock(Class), вы можете аннотировать поля вашего тестового класса.

Рассмотрим очередной пример с интерфейсом `DataService`:
``` java
public interface DataService {
    int retrieveData();
}
```
Мы можем создать мок DataService с использованием @Mock:
``` java

public class DataServiceTest {

    @Mock
    DataService mockDataService;

    @Before
    public void setUp() {
        MockitoAnnotations.initMocks(this);
    }

    @Test
    public void testRetrieveData() {
        Mockito.when(mockDataService.retrieveData()).thenReturn(5);
        assertEquals(5, mockDataService.retrieveData());
    }
}
```
@InjectMocks автоматически вставляет мок-объекты в тестируемый объект. Это удобно, когда у вас есть сложные зависимости.

Предположим, у нас есть класс DataProcessor, который зависит от DataService:
``` java
public class DataProcessor {
    private DataService dataService;

    public DataProcessor(DataService dataService) {
        this.dataService = dataService;
    }

    public int process() {
        return dataService.retrieveData() * 2;
    }
}
```
Мы можем использовать @InjectMocks для автоматического внедрения мока DataService в DataProcessor:

``` java
public class DataProcessorTest {
```

**@RunWith(MockitoJUnitRunner.class)**
@RunWith(MockitoJUnitRunner.class) - это аннотация JUnit, которая указывает, что тесты в классе должны запускаться с помощью специального "раннера" - MockitoJUnitRunner. Этот раннер инициализирует поля, аннотированные как @Mock, @Spy, @InjectMocks, автоматически, без необходимости вызывать MockitoAnnotations.initMocks(this) в каждом тестовом классе.

Допустим, у нас есть следующие классы:
``` java
public interface DataService {
    int retrieveData();
}

public class DataProcessor {
    private DataService dataService;

    public DataProcessor(DataService dataService) {
        this.dataService = dataService;
    }

    public int process() {
        return dataService.retrieveData() * 2;
    }
}
```
Наша цель - протестировать DataProcessor, используя мок DataService.

Для этого мы используем аннотацию @RunWith(MockitoJUnitRunner.class):
``` java
import org.junit.Test;
import org.junit.runner.RunWith;
import org.mockito.Mock;
import org.mockito.junit.MockitoJUnitRunner;
import static org.junit.Assert.assertEquals;
import static org.mockito.Mockito.when;

@RunWith(MockitoJUnitRunner.class)
public class DataProcessorTest {

    @Mock
    DataService mockDataService;

    @Test
    public void testProcess() {
        when(mockDataService.retrieveData()).thenReturn(5);

        DataProcessor dataProcessor = new DataProcessor(mockDataService);

        assertEquals(10, dataProcessor.process());
    }
}
```
Здесь MockitoJUnitRunner автоматически инициализирует mockDataService. Вам не нужно вызывать MockitoAnnotations.initMocks(this) в методе @Before.
**@MockBean - Интеграция Spring и Mockito**
Теоретические основы:
@MockBean - это аннотация Spring Boot Test, которая объединяет возможности Spring DI и Mockito.

Архитектурные аспекты:

Интегрируется с Spring TestContext Framework

Автоматически заменяет бины в ApplicationContext на мок-версии

Обеспечивает согласованность состояния между тестами

Отличия от @Mock:

- @Mock создает изолированный мок-объект

- @MockBean регистрирует мок в Spring контексте

- @MockBean обеспечивает инъекцию моков в автовайред зависимости

## Логирование в Spring Boot - Теоретический обзор
1. Архитектура логирования в Spring Boot
**Теоретическая основа:**
Spring Boot использует унифицированный подход к логированию, основанный на Spring Framework и реализации `SLF4J`. Ключевой принцип - **"opinionated defaults"** (продуманные настройки по умолчанию) с возможностью полной кастомизации.

**Компоненты архитектуры:**
`SLF4J API` - уровень абстракции:

- Единый интерфейс для всего приложения
- Независимость от конкретной реализации
- Согласованное API для разработчиков

Logback - реализация по умолчанию:
- Нативная реализация для SLF4J
- Высокая производительность
- Богатые возможности конфигурации

Мосты логирования (Bridges):
- Автоматическое перенаправление логов от других фреймворков (JUL, Log4j, Commons Logging)
- Единый канал логирования для всех компонентов приложения

2. Иерархия логгеров и уровни логирования
**Теоретическая модель:**
Иерархическая система логгеров наследует настройки от родительских логгеров. Корневой логгер - основа всей иерархии.

Уровни логирования (в порядке убывания важности):

- `ERROR` - критические ошибки, требующие немедленного внимания
- `WARN` - потенциально проблемные ситуации
- `INFO` - информационные сообщения о работе приложения
- `DEBUG` - детальная информация для разработки
- `TRACE` - максимально детальная трассировка выполнения

**Принцип наследования настроек:**
Логгеры наследуют уровень логирования и аппендеры от родительских логгеров, если не заданы явные настройки. Это позволяет централизованно управлять логированием для пакетов и компонентов.

3. Автоконфигурация логирования
**Теория автоматической настройки:**
Spring Boot применяет автоконфигурацию для логирования на основе:

- Наличие библиотек в classpath
- Файлы конфигурации в определенных местах
- Настройки в application.properties/yml
- Активные Spring профили

Приоритеты конфигурации:
- logback-spring.xml - Spring-специфичная конфигурация
- logback.xml - стандартная конфигурация Logback
- application.properties - базовые настройки
- Автоконфигурация - настройки по умолчанию

Spring-специфичные возможности:
Условная конфигурация через `<springProfile>` позволяет настраивать логирование для разных сред (dev, prod, test).

Spring Properties в конфигурации XML позволяют использовать значения из application.properties.

4. Управление логированием через свойства
**Теоретические аспекты:**
Централизованное управление через `application.properties`обеспечивает:

- Единую точку конфигурации
- Профилирование для разных сред
- Динамическое изменение уровней

**Принципы именования логгеров:**
- По пакетам: logging.level.com.example.myapp
- По классам: logging.level.com.example.MyService
- По компонентам Spring: logging.level.org.springframework

**Модель применения настроек:**
Настройки применяются каскадно - от более специфичных к более общим. Это позволяет тонко настраивать логирование для отдельных компонентов при сохранении общих правил.

1. Structured Logging (структурированное логирование)
**Теоретическая основа:**
Structured Logging - подход, при котором логи представляются в структурированном формате (JSON) вместо неструктурированного текста.

Преимущества структурированного логирования:

- Легкость парсинга и анализа
- Интеграция с системами мониторинга (ELK Stack, Splunk)
- Возможность агрегации и визуализации
- Улучшенный поиск и фильтрация

**Ключевые концепции:**
`MDC` (Mapped Diagnostic Context) - механизм для добавления контекстной информации ко всем логам в пределах потока выполнения.

Корреляционные идентификаторы - уникальные ID для отслеживания запросов через распределенные системы.

6. Производительность логирования
**Теоретические аспекты производительности:**
Cost of Logging - влияние логирования на производительность приложения:
- Создание строковых представлений объектов
- Время выполнения условных проверок
- Накладные расходы на ввод-вывод

Parameterized Logging - использование шаблонов вместо конкатенации строк:
- Избегает создания строк при отключенном уровне логирования
- Значительно улучшает производительность

Принципы эффективного логирования:
- Проверка уровня логирования перед созданием сложных сообщений
- Использование Supplier для ленивого вычисления сообщений
- Минимизация логирования в горячих путях выполнения

7. Безопасность и логирование
**Теоретические аспекты безопасности:**
Чувствительные данные в логах:
- Пароли, токены, персональные данные
- Финансовая информация
- Конфиденциальные метаданные

Принципы безопасного логирования:
- Маскирование чувствительных полей
- Контекстно-зависимое логирование
- Аудит логов на наличие утечек

8. Мониторинг и управление логированием
**Теория динамического управления:**
Actuator Endpoints предоставляют REST API для управления логированием во время выполнения:
- Просмотр и изменение уровней логирования
- Просмотр конфигурации логгеров
- Мониторинг состояния логирования

Принципы операционного управления:
- Журналирование событий (Audit Logging) - специальный тип логирования для отслеживания значимых бизнес-событий и действий пользователей.
- Ротация лог-файлов - автоматическое управление размером и возрастом лог-файлов для предотвращения переполнения диска.

9. Интеграция с облачными платформами
**Теоретические основы:**
Cloud-Native Logging - подходы к логированию в облачных средах:

- Логирование в stdout/stderr
- Интеграция с системными демонами (journald)
- Форматирование для облачных провайдеров

Distributed Tracing - сквозная трассировка запросов в микросервисной архитектуре:
- Корреляция логов между сервисами
- Визуализация путей выполнения
- Анализ производительности

**Базовое использование в сервисе**
``` java

        }
```

1. Интеграционная архитектура тестирования
**Теория тестовых пирамид:**
Классическая пирамида тестирования:

- Unit tests (основа) - быстрые, изолированные тесты
- Integration tests (середина) - тестирование взаимодействий
- End-to-end tests (вершина) - полные сценарии работы системы

``` text
    ╱╲       E2E Tests (5-10%)
   ╱  ╲      Конечные сценарии, проверка работы всей системы
  ╱____╲     
 ╱──────╲    Integration Tests (15-20%)
╱────────╲   Проверка взаимодействия компонентов
──────────   Unit Tests (70-80%)
             Изолированная проверка отдельных модулей
```
Unit Tests (Модульные тесты)
**Теоретическая основа:**
Unit тесты проверяют отдельные модули или компоненты в полной изоляции от внешних зависимостей.

**Ключевые характеристики:**
- Скорость: Выполняются за миллисекунды
- Изоляция: Полная изоляция от внешних зависимостей
- Детерминированность: Предсказуемый результат
- Фокус: Одна ответственность (Single Responsibility)

Когда использовать моки в unit тестах:
Обязательное мокирование:
- Внешние сервисы (API, базы данных, message brokers)
- Сложные зависимости с побочными эффектами
- Компоненты с недетерминированным поведением
- Ресурсоемкие операции

Избегать мокирования:
- Value Objects (DTO, Entities без поведения)
- Pure functions (детерминированные функции без side effects)
- Внутренние вспомогательные методы

Пример unit теста:
``` java
// Тестируемый сервис
@Service
public class OrderService {
    private final PaymentService paymentService;
    private final InventoryService inventoryService;
    
    public Order processOrder(Order order) {
        if (!inventoryService.checkAvailability(order.getItems())) {
            throw new InventoryException("Not enough inventory");
        }
        
        PaymentResult result = paymentService.processPayment(order);
        if (!result.isSuccess()) {
            throw new PaymentException("Payment failed");
        }
        
        return order.confirm();
    }
}

// Unit test
class OrderServiceTest {
    @Test
    void processOrder_WithAvailableInventory_ShouldProcessPayment() {
        // Arrange
        PaymentService paymentService = mock(PaymentService.class);
        InventoryService inventoryService = mock(InventoryService.class);
        OrderService orderService = new OrderService(paymentService, inventoryService);
        
        Order order = new Order(SAMPLE_ITEMS);
        when(inventoryService.checkAvailability(order.getItems())).thenReturn(true);
        when(paymentService.processPayment(order)).thenReturn(new PaymentResult(true));
        
        // Act
        Order result = orderService.processOrder(order);
        
        // Assert
        assertEquals(OrderStatus.CONFIRMED, result.getStatus());
        verify(paymentService).processPayment(order);
    }
}
```
3. Integration Tests (Интеграционные тесты)
**Теоретическая основа:**
Integration тесты проверяют взаимодействие между несколькими компонентами системы.
**Ключевые характеристики:**
- Объем: Проверяют группы связанных компонентов
- Зависимости: Используют реальные или приближенные к реальным зависимости
- Скорость: Выполняются дольше unit тестов
- Фокус: Взаимодействие и интеграция

**Стратегии мокирования в integration тестах:**
Частичное мокирование:
- Только внешние системы (сторонние API, платежные шлюзы)
- Компоненты с длительным временем выполнения
- Сервисы, требующие специального окружения

Реальные зависимости:
- Внутренние сервисы приложения
- Базы данных (используя test containers или in-memory DB)
- Message brokers (если возможно использовать embedded версии)

Пример integration теста:
``` java
@SpringBootTest
class OrderIntegrationTest {
    @Autowired
    private OrderService orderService;
    
    @MockBean
    private ExternalPaymentGateway paymentGateway; // Мок внешнего сервиса
    
    @Autowired
    private OrderRepository orderRepository; // Реальный репозиторий
    
    @Test
    void completeOrderFlow_WithRealDatabase_ShouldWork() {
        // Arrange
        Order order = orderRepository.save(createTestOrder());
        when(paymentGateway.process(any())).thenReturn(PaymentStatus.SUCCESS);
        
        // Act
        Order result = orderService.processOrder(order.getId());
        
        // Assert
        assertNotNull(result.getConfirmedAt());
        assertEquals(OrderStatus.COMPLETED, result.getStatus());
    }
}
```
4. Стратегии использования моков: Глубокий анализ
**Принципы эффективного мокирования:**
Закон Деметры для тестов:
- Мокайте только непосредственные зависимости
- Избегайте цепочек моков (mock returning mock)
- Фокусируйтесь на контрактах, а не на реализации

Black-box vs White-box testing:
- Black-box: Мокирование на основе публичных контрактов
- White-box: Мокирование с учетом внутренней реализации (осторожно!)

Типичные антипаттерны мокирования:
Over-mocking:

``` java
// ПЛОХО: Избыточное мокирование
@Test
void antiPattern_OverMocking() {
    User user = mock(User.class);
    when(user.getId()).thenReturn(1L);
    when(user.getName()).thenReturn("John");
    when(user.getEmail()).thenReturn("john@test.com");
    when(user.isActive()).thenReturn(true);
    // ... и так далее
    
    // Лучше использовать реальный объект с тестовыми данными
    User user = new User(1L, "John", "john@test.com", true);
}
Mocking value objects:

java
// ПЛОХО: Мокирование простых объектов
@Test
void antiPattern_MockingValueObject() {
    Address address = mock(Address.class);
    when(address.getCity()).thenReturn("Moscow");
    when(address.getStreet()).thenReturn("Lenina");
    
    // Лучше использовать builder pattern
    Address address = Address.builder()
        .city("Moscow")
        .street("Lenina")
        .build();
}
```

5. Критерии выбора типа тестов
Decision Matrix:
Критерий|Unit Tests|Integration Tests
---|---|---
Скорость выполнения|Высокая|Средняя-низкая
Стоимость поддержки|Низкая|Средняя-высокая
Coverage scope|Один компонент|Группа компонентов
Требования к изоляции|Полная изоляция|Контролируемые зависимости
Частота выполнения|При каждом коммите|Ночные сборки

**Когда что выбирать:**
Unit tests идеальны для:
- Алгоритмов и бизнес-логики
- Трансформаций данных
- Валидации и проверок
- Pure functions

Integration tests необходимы для:
- Работы с базой данных
- Внешних API интеграций
- Message-driven архитектур
- Spring контекста и DI

6. Продвинутые стратегии
Test Doubles Spectrum:
**Test Double** - это общий термин для любых объектов, которые заменяют реальные компоненты в тестовой среде. Концепция происходит из кинематографа, где "дублеры" заменяют актеров в опасных сценах.
От простого к сложному:
- Dummy - пустышка, никогда не используется
``` java
// Реальный сервис
public class NotificationService {
    public void sendEmail(String to, EmailTemplate template) {
        // Реальная логика отправки email
    }
}

// В тесте - Dummy объект
@Test
void userRegistration_ShouldCreateUser_WithDummyNotification() {
    // Arrange
    EmailTemplate dummyTemplate = null; // Dummy - не используется в тесте
    UserService userService = new UserService();
    
    // Act
    User user = userService.register("user@test.com", dummyTemplate);
    
    // Assert
```

Contract Testing альтернатива:
**Contract Testing** - это методика тестирования, которая проверяет, что взаимодействие между сервисами (consumer и provider) соответствует ожидаемому контракту.
Проблема, которую решает Contract Testing:
Integration Hell в микросервисной архитектуре:
- Сервисы развиваются независимо
- Ломающие изменения в API
- Сложность энд-to-энд тестирования
  - Ложные срабатывания в тестах

1. Теоретические паттерны тестирования
**Паттерны организации тестов:**
Arrange-Act-Assert (AAA):
- Arrange - настройка тестового окружения
- Act - выполнение тестируемого действия
- Assert - проверка ожидаемых результатов

Given-When-Then:
- Более описательный вариант AAA
- Часто используется в BDD (Behavior-Driven Development)

**Паттерны мокирования:**
Spy vs Mock:
- Mock - полностью фиктивный объект с запрограммированным поведением
- Spy - обертка вокруг реального объекта с возможностью переопределения методов

Stubbing vs Verification:
- Stubbing - определение поведения мок-объекта
- Verification - проверка взаимодействий с мок-объектом

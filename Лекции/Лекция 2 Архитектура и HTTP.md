 1. Клиент-серверная архитектура
Что это?
Модель взаимодействия, где:

Клиент — инициатор запроса (браузер, мобильное приложение, curl и т.д.).
Сервер — обработчик запроса, возвращающий данные (веб-сервер, API, база данных и т.д.).
Принцип работы:
Клиент отправляет запрос.
Сервер его обрабатывает.
Сервер возвращает ответ.
Клиент отображает или использует ответ.
📌 Пример: Вы вводите https://google.com → браузер (клиент) запрашивает HTML-страницу → сервер Google возвращает её → браузер отрисовывает страницу.

📡 2. HTTP — протокол передачи гипертекста
HTTP (HyperText Transfer Protocol) — прикладной протокол для передачи данных в клиент-серверной архитектуре.

Основные методы HTTP:
Метод|Описание|Идемпотентность|Безопасность
---|---|---|---
GET|Получение ресурса (не меняет состояние сервера)|✅ Да|✅ Да
POST|Создание ресурса (например, отправка формы)|❌ Нет|❌ Нет
PUT|Полная замена ресурса (если нет — создаст)|✅ Да|❌ Нет
DELETE|Удаление ресурса|✅ Да|❌ Нет

🔹 Идемпотентность — повторный вызов не меняет результат (например, удалить файл дважды — всё равно будет удалён один раз).
🔹 Безопасность — метод не должен изменять состояние сервера.

📌 Пример:

GET /users → получить список пользователей
POST /users → создать нового пользователя
PUT /users/123 → обновить пользователя с ID 123
DELETE /users/123 → удалить пользователя с ID 123
📊 3. HTTP Статус-коды
Сервер отвечает не только данными, но и кодом состояния — трёхзначным числом, показывающим результат обработки запроса.

Основные группы:
Группа|Значение|Примеры
---|---|---
1xx|Информационные|100 Continue
2xx|Успех|200 OK, 201 Created
3xx|Перенаправления|301 Moved Permanently, 304 Not Modified
4xx|Ошибки клиента|400 Bad Request, 404 Not Found, 403 Forbidden
5xx|Ошибки сервера|500 Internal Server Error, 502 Bad Gateway

📌 Пример:

404 — вы запросили /page-that-does-not-exist — сервер не нашёл ресурс.
500 — сервер сломался при обработке вашего запроса.
🌐 4. TCP/IP — фундамент интернета
HTTP работает поверх TCP/IP. Чтобы понять, почему запросы иногда медленные — нужно заглянуть глубже.

🔁 TCP Handshake (трёхэтапное рукопожатие)
Перед отправкой данных клиент и сервер договариваются о соединении:

SYN — клиент: “Привет, можно поговорить?”
SYN-ACK — сервер: “Привет, да, давай!”
ACK — клиент: “Окей, начинаем!”
⏱️ Это добавляет задержку! → Минимум 1 RTT (Round-Trip Time) перед отправкой первого байта данных.

🔄 Retransmission (повторная передача)
TCP гарантирует доставку. Если пакет потерялся — он будет отправлен снова. Это хорошо для надёжности, но плохо для скорости при высоких потерях.

📏 Bandwidth-Delay Product (BDP)
BDP = Пропускная способность × Задержка 

📌 Что это значит?
— Это максимальное количество данных, которое может быть "в пути" между клиентом и сервером в любой момент времени.

🔹 Пример:

Пропускная способность: 10 Мбит/с = 1.25 МБ/с
Задержка (RTT): 200 мс = 0.2 с
→ BDP = 1.25 × 0.2 = 0.25 МБ
💡 Зачем это знать?
Если окно TCP меньше BDP — канал используется неэффективно. Сервер будет ждать подтверждения, прежде чем отправить следующую порцию данных → снижение скорости.

🚧 5. Flow Control и Head-of-Line (HOL) Blocking
🌊 Flow Control (Управление потоком)
TCP использует механизм окон, чтобы не перегружать получателя. Сервер не отправит больше данных, чем клиент готов принять.

→ Это предотвращает переполнение буфера, но может ограничивать скорость, если клиент медленный.

🚫 Head-of-Line Blocking (Блокировка головы очереди)
Проблема: В HTTP/1.1 все запросы в одном TCP-соединении идут последовательно. Если первый запрос “завис” — все остальные ждут.

📌 Пример:
Вы загружаете страницу с 10 изображениями. Первое грузится 5 секунд → остальные 9 ждут эти 5 секунд, даже если они маленькие и быстрые!

🚀 6. Как HTTP/2 решает проблему HOL Blocking
HTTP/2 — это эволюция HTTP/1.1, построенная на тех же принципах, но с кардинальными улучшениями:

✅ Основные фичи:
Multiplexing (мультиплексирование)
— Несколько запросов и ответов могут передаваться одновременно по одному TCP-соединению.
— Нет очереди → нет HOL Blocking!
Header Compression (HPACK)
— Заголовки сжимаются → меньше данных → быстрее передача.
Server Push
— Сервер может “предугадать” и прислать ресурсы, которые клиент запросит позже (например, CSS/JS для HTML).
📌 Пример:
Теперь 10 изображений грузятся параллельно — даже если одно большое, остальные не ждут!

🔍 7. Практика: Анализ запросов через Chrome DevTools (Network Tab)
Давайте посмотрим всё это в действии!

Шаги:
Откройте любой сайт (например, https://example.com).
Нажмите F12 → вкладка Network.
Обновите страницу (Ctrl+R).
Посмотрите список запросов.
Что можно увидеть:
Method — GET, POST и т.д.
Status — 200, 404, 301...
Type — document, script, image, xhr...
Initiator — что вызвало запрос (HTML, JS и т.д.)
Waterfall — визуализация времени:
DNS Lookup
Initial connection (TCP handshake!)
SSL (если HTTPS)
Time to First Byte (TTFB)
Content Download
📌 Практическое задание:
Найдите запрос с долгим TTFB — возможно, сервер медленно обрабатывает запрос.
Найдите запрос с большим Content Download — возможно, тяжёлое изображение или файл.

🧩 Дополнительно: Почему важно понимать TCP для веб-разработчика?
Потому что:

Latency ≠ Bandwidth. Можно иметь гигабитный интернет, но если RTT = 300 мс — сайт будет “тормозить”.
Один TCP handshake = минимум 1 RTT. Если у вас 50 ресурсов — и каждый по отдельному соединению — вы потеряете 50 × RTT!
HTTP/2 экономит соединения → меньше handshake → быстрее загрузка.

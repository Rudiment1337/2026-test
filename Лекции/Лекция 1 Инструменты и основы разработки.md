# **Лекция 1: Инструменты и основы разработки**

## **1. VS Code (настройка Java, расширения)**

### Что такое VS Code?
**Visual Studio Code (VS Code)** — это бесплатный, кроссплатформенный редактор кода от Microsoft. Он поддерживает множество языков программирования, включая Java, и активно используется в разработке благодаря своей легковесности и мощной экосистеме расширений.

### Установка и настройка Java в VS Code

1. **Установите JDK (Java Development Kit)**  
   - Рекомендуется использовать **OpenJDK** (например, от Adoptium, Oracle, или Amazon Corretto).
   - Убедитесь, что переменная окружения `JAVA_HOME` указывает на папку с JDK.

2. **Установите VS Code**  
   - Скачайте с официального сайта: [https://code.visualstudio.com](https://code.visualstudio.com)

3. **Установите расширения для Java**  
   Основные расширения:
   - **Extension Pack for Java** (от Microsoft) — включает:
     - Language Support for Java
     - Debugger for Java
     - Test Runner for Java
     - Maven for Java
     - Project Manager for Java
     - Visual Studio IntelliCode
   - **Spring Boot Tools** — для работы с Spring Boot приложениями.
   - **Lombok** — если используется аннотации Lombok.

> 💡 **Пример**: После установки расширений, при открытии `.java` файла вы увидите автодополнение, подсветку ошибок, возможность запуска и отладки.

### Преимущества VS Code для Java:
- Быстрый запуск.
- Хорошая интеграция с Git.
- Поддержка Spring Boot.
- Лёгкий по сравнению с IDE вроде IntelliJ IDEA.

## **2. Maven vs Gradle (сравнение, структура проекта)**

### Общее назначение
Maven и Gradle — это **системы сборки (build tools)**, которые помогают:
- Управлять зависимостями.
- Компилировать код.
- Запускать тесты.
- Собирать JAR/WAR-файлы.
- Развертывать приложения.

---

### **Maven**

#### Файл конфигурации: `pom.xml`
- Основан на XML.
- Жёсткая структура проекта (конвенция над конфигурацией).
- Репозитории: Maven Central, JCenter (устарел), приватные.

#### Пример `pom.xml` (Spring Boot):
```xml
<?xml version="1.0" encoding="UTF-8"?>
<project xmlns="http://maven.apache.org/POM/4.0.0"
         xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
         xsi:schemaLocation="http://maven.apache.org/POM/4.0.0 
         http://maven.apache.org/xsd/maven-4.0.0.xsd">
    <modelVersion>4.0.0</modelVersion>
    <groupId>com.example</groupId>
    <artifactId>demo-app</artifactId>
    <version>1.0.0</version>
    <packaging>jar</packaging>

    <parent>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-parent</artifactId>
        <version>3.2.0</version>
        <relativePath/>
    </parent>

    <dependencies>
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-starter-web</artifactId>
        </dependency>
    </dependencies>

    <build>
        <plugins>
            <plugin>
                <groupId>org.springframework.boot</groupId>
                <artifactId>spring-boot-maven-plugin</artifactId>
            </plugin>
        </plugins>
    </build>
</project>
```

#### Плюсы Maven:
- Простота и стандартизация.
- Большое сообщество.
- Хорошая документация.

#### Минусы:
- Громоздкий XML.
- Медленнее сборка по сравнению с Gradle.
- Менее гибкий.

---

### **Gradle**

#### Файл конфигурации: `build.gradle` (на Groovy или Kotlin DSL)
- Использует **DSL** (Domain-Specific Language).
- Поддерживает Groovy и Kotlin (Kotlin DSL рекомендуется).
- Более гибкий и быстрый (инкрементная сборка, кэширование).

#### Пример `build.gradle` (на Groovy):
```groovy
plugins {
    id 'java'
    id 'org.springframework.boot' version '3.2.0'
    id 'io.spring.dependency-management' version '1.1.3'
}

group = 'com.example'
version = '1.0.0'

repositories {
    mavenCentral()
}

dependencies {
    implementation 'org.springframework.boot:spring-boot-starter-web'
    testImplementation 'org.springframework.boot:spring-boot-starter-test'
}
```

#### Плюсы Gradle:
- Более читаемый синтаксис.
- Высокая производительность.
- Гибкость (можно писать кастомные задачи).
- Поддержка Kotlin DSL.

#### Минусы:
- Сложнее для новичков.
- Меньше стандартов по сравнению с Maven.



### **Структура типичного Java-проекта**

```
src/
├── main/
│   ├── java/              → Java-классы
│   │   └── com/example/
│   │       └── App.java
│   ├── resources/         → Конфиги, файлы (application.yml, static/)
│   │   └── application.yml
│   └── webapp/            → JSP (если есть)
├── test/
│   ├── java/              → Тесты
│   └── resources/         → Тестовые ресурсы
pom.xml                    → Maven
build.gradle               → Gradle
settings.gradle            → Gradle (мультимодульные проекты)
```

## **3. XML / JSON / YAML — форматы данных**

### **XML (eXtensible Markup Language)**

- Иерархическая структура с тегами.
- Читаемый, но многословный.
- Используется в Maven (`pom.xml`), Spring (ранние версии), конфигурации серверов (Tomcat, web.xml).

#### Пример:
```xml
<user>
    <id>1</id>
    <name>John</name>
    <active>true</active>
</user>
```

### **JSON (JavaScript Object Notation)**

- Легковесный формат обмена данными.
- Основан на JavaScript.
- Используется в REST API, конфигах (редко), NoSQL БД.

#### Пример:
```json
{
  "id": 1,
  "name": "John",
  "active": true
}
```

### **YAML (YAML Ain't Markup Language)**

- Читаемый формат с отступами (как Python).
- Используется в Spring Boot (`application.yml`), Docker Compose, Kubernetes.

#### Пример:
```yaml
user:
  id: 1
  name: John
  active: true
```

## **4. Фреймворк vs Библиотека (на примере Spring и Apache Commons)**

### **Библиотека (Library)**

- Это **набор готовых методов и классов**, которые **вы вызываете**.
- Вы управляете потоком выполнения.
- Пример: `Apache Commons Lang`, `Google Guava`, `Jackson`.

#### Пример с Apache Commons:
```java
import org.apache.commons.lang3.StringUtils;

public class Main {
    public static void main(String[] args) {
        String text = "  Hello  ";
        String trimmed = StringUtils.trim(text); // Вы сами решаете, когда вызывать
        System.out.println(trimmed);
    }
}
```

- Вы **вызываете** `StringUtils.trim()`.
- Вы контролируете выполнение.

---

### **Фреймворк (Framework)**

- Это **основа приложения**, которая **вызывает ваш код**.
- Принцип: **"Не мы вызываем фреймворк, фреймворк вызывает нас"** (Inversion of Control).
- Пример: **Spring Framework**, Hibernate, Struts.

#### Пример с Spring Boot:
```java
@RestController
public class UserController {

    @GetMapping("/hello")
    public String hello() {
        return "Hello, World!";
    }
}
```

- Вы не вызываете `hello()` напрямую.
- Spring **сам находит** этот метод по аннотации `@RestController` и `@GetMapping`.
- Spring **управляет жизненным циклом** бинов, зависимостями, маршрутизацией.

---

### Ключевое различие:

| Критерий               | Библиотека                          | Фреймворк                              |
|------------------------|-------------------------------------|-----------------------------------------|
| Контроль выполнения    | У разработчика                     | У фреймворка (IoC)                       |
| Как используется       | Вызов методов                       | Аннотации, конфигурация, интеграция     |
| Пример                 | Apache Commons, Jackson             | Spring, Hibernate, React                |
| Гибкость               | Высокая (можно использовать частично) | Низкая (всё или почти всё)              |
| Зависимость            | Лёгкая                              | Глубокая (проект строится вокруг него)  |

## **Дополнение: Системы контроля версий — Git, GitHub, GitLab**

### **1. Что такое Git?**

**Git** — это **распределённая система контроля версий (DVCS)**, разработанная Линусом Торвальдсом в 2005 году. Она позволяет:
- Отслеживать изменения в файлах (в основном в коде).
- Работать в команде без конфликтов.
- Возвращаться к предыдущим версиям.
- Создавать ветки для новых функций и экспериментов.

#### 🔹 Основные понятия Git:

| Термин       | Описание |
|--------------|--------|
| **Репозиторий (repo)** | Папка с проектом, где ведётся контроль версий. Содержит скрытую папку `.git`. |
| **Коммит (commit)** | "Снимок" состояния файлов в определённый момент. Каждый коммит имеет уникальный хеш (например, `a1b2c3d`). |
| **Ветка (branch)** | Независимая линия разработки. По умолчанию — `main` или `master`. |
| **Слияние (merge)** | Объединение изменений из одной ветки в другую. |
| **Клонирование (clone)** | Создание локальной копии удалённого репозитория. |
| **Push / Pull** | `push` — отправка своих коммитов на сервер; `pull` — получение изменений от других. |
| **Staging area (index)** | Промежуточная зона, куда добавляются изменения перед коммитом (`git add`). |

---

### **2. Основные команды Git**

```bash
# Инициализация репозитория
git init

# Клонирование репозитория
git clone https://github.com/user/project.git

# Проверка статуса (какие файлы изменены)
git status

# Добавить файлы в staging
git add filename.java
git add .  # все изменения

# Создать коммит
git commit -m "Описание изменений"

# Отправить изменения на сервер
git push origin main

# Получить последние изменения
git pull origin main

# Создать новую ветку
git branch feature/login

# Переключиться на ветку
git checkout feature/login
# или короткая версия:
git switch feature/login

# Создать и переключиться на ветку
git checkout -b feature/login

# Слить ветку в текущую (например, в main)
git merge feature/login

# Посмотреть историю коммитов
git log --oneline
```

### **3. Рабочий процесс с ветками (Branching Workflow)**

Современные команды используют **ветвистые стратегии**, например:
- **Feature Branch Workflow**
- **Git Flow**
- **GitHub Flow**

#### Пример: Feature Branch Workflow
1. Вы создаете ветку от `main`: `git checkout -b feature/user-auth`
2. Пишете код, делаете коммиты.
3. Push: `git push origin feature/user-auth`
4. На GitHub/GitLab создаёте **Pull Request (PR)** или **Merge Request (MR)**.
5. Коллеги проверяют код (code review).
6. После одобрения — вливаете в `main`.

> 💡 Это предотвращает прямое изменение основной ветки и повышает качество кода.

### **4. GitHub vs GitLab — что это?**

Оба — **хостинговые платформы для Git-репозиториев**, но с дополнительными возможностями.

| Характеристика       | **GitHub**                            | **GitLab**                             |
|----------------------|----------------------------------------|-----------------------------------------|
| Владелец             | Microsoft                              | GitLab Inc.                             |
| Основной фокус       | Open Source, разработка ПО             | DevOps, CI/CD, полный цикл разработки   |
| CI/CD                | GitHub Actions                         | GitLab CI/CD (встроен, мощный)          |
| Самостоятельная установка | Можно (GitHub Enterprise)         | Да (GitLab CE/EE — бесплатная версия)   |
| Интеграции           | Очень много (VS Code, Slack, Docker)   | Хорошие, но меньше экосистемы           |
| Pull Request         | ✅                                     | ✅ (называется Merge Request)            |
| Wiki, Issues, Pages  | ✅                                     | ✅                                       |
| Бесплатные приватные репозитории | ✅ (с ограничениями)        | ✅                                       |

### **7. Полезные практики при работе с Git**

1. **Пишите понятные сообщения коммитов**  
   Плохо: `fix bug`  
   Хорошо: `fix: prevent crash when user submits empty form`

2. **Используйте `.gitignore`**  
   Файл, в котором перечислены папки/файлы, которые **не нужно добавлять в Git**:
   ```
   target/
   build/
   .idea/
   *.log
   .env
   ```

3. **Регулярно обновляйте ветку `main`**  
   ```bash
   git checkout main
   git pull origin main
   ```

4. **Не коммитьте чувствительные данные** (пароли, ключи API).  
   Используйте `.env` + `.gitignore` + переменные окружения.

5. **Работайте в ветках, а не в `main` напрямую.**

### **Итог: Почему Git обязателен?**

| Причина | Объяснение |
|--------|-----------|
| **История изменений** | Можно откатиться к любой версии. |
| **Командная работа** | Несколько человек могут работать параллельно. |
| **Резервное копирование** | Код хранится не только локально, но и на сервере. |
| **CI/CD** | Автоматические тесты, сборка, деплой. |
| **Code Review** | Коллеги проверяют код до слияния. |

# Лекция 4: Веб-серверы и многопоточность

## Цель лекции
Научиться понимать, как веб-серверы в Spring Boot обрабатывают запросы, какие модели многопоточности используются, и как избежать типичных ошибок при работе с общими данными в многопоточной среде.

---

## 1. Веб-серверы в Spring Boot: Tomcat и Jetty

Spring Boot по умолчанию использует **встроенный веб-сервер**. Самый распространённый — **Apache Tomcat**, который подключается через зависимость `spring-boot-starter-web`.

Также можно использовать:
- **Jetty** — легковесный, подходит для микросервисов.
- **Undertow** — высокопроизводительный, от Red Hat.

> Сервер запускается при старте приложения и принимает HTTP-запросы.

Каждый запрос обрабатывается **в отдельном потоке** из пула — это позволяет серверу обслуживать множество клиентов одновременно.

---

## 2. Как работает обработка запроса?

Когда клиент делает HTTP-запрос:

1. Сервер (Tomcat/Jetty) получает соединение.
2. Берёт свободный поток из **пула потоков**.
3. Вызывает соответствующий метод контроллера (`@GetMapping`, `@PostMapping` и т.д.).
4. Метод выполняется, возвращает результат.
5. Ответ отправляется клиенту.

> ⚠️ Все контроллеры в Spring — **синглтоны**. Это значит, что один и тот же экземпляр класса может одновременно обрабатывать запросы в нескольких потоках.

---

## 3. Модели многопоточности

### One Thread per Connection (устарело)
- На каждый запрос создаётся новый поток.
- Неэффективно: много потоков → высокое потребление памяти.

### Thread Pool (современный подход)
- Заранее создан пул потоков (например, 10–200).
- Запросы обслуживаются свободными потоками из пула.
- Используется в Tomcat, Jetty и других серверах.

#### Настройка пула в Spring Boot (`application.properties`)
```properties
# Для Tomcat
server.tomcat.threads.max=50
server.tomcat.threads.min-spare=5

# Максимальное число соединений
server.tomcat.max-connections=1000
```
### 4. Пример гонки за данными (Race Condition)
- Допустим, мы хотим посчитать, сколько раз был вызван эндпоинт.

- ❌ Неправильно: обычная переменная
``` java
@RestController
public class CounterController {

    private int requestCount = 0;

    @GetMapping("/counter")
    public String increment() {
        requestCount++; // <-- НЕПОТОКОБЕЗОПАСНО!
        return "Запрос №" + requestCount;
    }
}
```

> Почему опасно? 

- requestCount++ — это три операции: чтение, инкремент, запись.
- Если два потока прочитают значение одновременно, оба увеличат его и запишут — один результат будет потерян.
### 5. Демонстрация проблемы
Запустите 1000 параллельных запросов:
``` bash
for i in {1..1000}; do curl http://localhost:8080/counter & done
```
- ➡️ Ожидаемо: Запрос №1000
- ➡️ На практике: часто Запрос №996, 993 и т.д.

Вывод: состояние гонки реально и приводит к ошибкам.

### 6. Решение 1: synchronized
Вариант A: синхронизация метода
``` java
@GetMapping("/counter")
public synchronized String increment() {
    requestCount++;
    return "Запрос №" + requestCount;
}
```
✅ Потокобезопасно.
❌ Все запросы теперь ждут друг друга — снижается производительность.

Вариант B: синхронизированный блок
``` java
private final Object lock = new Object();

@GetMapping("/counter")
public String increment() {
    synchronized (lock) {
        requestCount++;
    }
    return "Запрос №" + requestCount;
}
```
✅ Только критическая секция блокируется.
✅ Лучше по производительности.

### 7. Решение 2: AtomicInteger (рекомендуется)
Используйте атомарные типы из java.util.concurrent.atomic.
``` java
import java.util.concurrent.atomic.AtomicInteger;

@RestController
public class CounterController {

    private final AtomicInteger requestCount = new AtomicInteger(0);

    @GetMapping("/counter")
    public String increment() {
        int current = requestCount.incrementAndGet(); // атомарно +1
        return "Запрос №" + current;
    }
}
```
✅ Потокобезопасно
✅ Без блокировок (lock-free)
✅ Высокая производительность
✅ Идеально для счётчиков

### 8. ExecutorService в Spring
Можно создать свой пул потоков для фоновых задач.
``` java
@Configuration
public class ThreadPoolConfig {

    @Bean
    public ExecutorService customExecutor() {
        return Executors.newFixedThreadPool(10);
    }
}
```
Использование:


``` java 
@Service
public class BackgroundService {

    @Autowired
    private ExecutorService executor;

    public void runTask(Runnable task) {
        executor.submit(task);
    }
}
```
💡 Альтернатива: аннотация @Async (требует @EnableAsync).

9. Amdahl’s Law: почему параллельность не решает всё
> "Ускорение системы ограничено долей последовательного кода." 

Формула:
\[
S = \frac{1}{(1 - P) + \frac{P}{N}}
\]
​
Где:

- S — максимальное ускорение,
- P — доля параллелизуемого кода,
- N — число ядер.
Пример
- Если 90% кода можно распараллелить:

- На 10 ядрах: ускорение ≈ 5.26×
- На бесконечном числе ядер: максимум 10×!
> Даже если вы добавите 1000 потоков, узкое место (например, база данных или синхронизация) ограничит рост производительности.

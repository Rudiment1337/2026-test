# Лекция 8: Запросы и оптимизация

## 1. Query Methods (Методы запросов)
**Теоретический материал:**
**Query Methods** — это декларативный способ создания запросов в Spring Data JPA путем именования методов репозитория по определенным правилам. Spring автоматически генерирует JPQL/SQL запросы на основе имени метода.

**Синтаксис и правила именования:**
``` text
[action]By[property][condition][And/Or][property2][condition2]...
```
**Основные ключевые слова:**

- `findBy`, `readBy`, `queryBy`, `getBy` — для выборки

- `countBy` — для подсчета

- `deleteBy`, `removeBy` — для удаления

- `existsBy` — для проверки существования

**Практические примеры:**

``` java 
public interface UserRepository extends JpaRepository<User, Long> {
    
    // Простые условия
    List<User> findByName(String name);
    Optional<User> findByEmail(String email);
    
    // Комбинированные условия
    List<User> findByNameAndEmail(String name, String email);
    List<User> findByNameOrEmail(String name, String email);
    
    // Операторы сравнения
    List<User> findByAgeGreaterThan(int age);
    List<User> findByAgeLessThanEqual(int age);
    List<User> findByAgeBetween(int minAge, int maxAge);
    
    // Строковые операции
    List<User> findByNameContaining(String part);
    List<User> findByNameLike(String pattern); // где pattern = "%john%"
    List<User> findByNameStartingWith(String prefix);
    List<User> findByNameEndingWith(String suffix);
    List<User> findByNameIgnoreCase(String name);
    
    // Коллекции
    List<User> findByAgeIn(List<Integer> ages);
    List<User> findByRolesEmpty();
    
    // Сортировка
    List<User> findByActiveTrueOrderByNameAsc();
    List<User> findByAgeGreaterThanOrderByAgeDescNameAsc(int age);
    
    // Ограничение результатов
    User findFirstByOrderByAgeDesc();
    List<User> findTop5ByOrderByCreatedDateDesc();
    
    // Подсчет и проверка
    long countByActiveTrue();
    boolean existsByEmail(String email);
}
```

**Когда использовать:**
- ✅ Для простых запросов

- ✅ Когда условия можно выразить через именование

- ✅ Для быстрой разработки

**Когда не использовать:**
- ❌ Сложные JOINы с несколькими таблицами

- ❌ Сложная бизнес-логика в запросе

- ❌ Когда нужна максимальная производительность

## 2. @Query (Кастомные запросы)
**Теоретический материал:**
**@Query** позволяет писать собственные JPQL или нативные SQL запросы, когда Query Methods недостаточно.

**Практические примеры:**
``` java
public interface UserRepository extends JpaRepository<User, Long> {
    
    // JPQL запрос (рекомендуется)
    @Query("SELECT u FROM User u WHERE u.age > :minAge AND u.active = true")
    List<User> findActiveUsersOlderThan(@Param("minAge") int minAge);
    
    // JPQL с JOIN
    @Query("SELECT u FROM User u JOIN u.orders o WHERE o.totalAmount > :amount")
    List<User> findUsersWithLargeOrders(@Param("amount") BigDecimal amount);
    
    // Проекция (возврат DTO вместо Entity)
    @Query("SELECT new com.example.UserDTO(u.id, u.name, u.email) FROM User u WHERE u.active = true")
    List<UserDTO> findActiveUsersAsDTO();
    
    // Нативный SQL запрос (используйте только когда JPQL недостаточно)
    @Query(value = "SELECT * FROM users u WHERE u.created_date > :date", nativeQuery = true)
    List<User> findUsersCreatedAfter(@Param("date") Date date);
    
    // Модифицирующие запросы (требует @Modifying)
    @Modifying
    @Query("UPDATE User u SET u.active = false WHERE u.lastLoginDate < :date")
    int deactivateInactiveUsers(@Param("date") LocalDateTime date);
    
    // Сложный нативный запрос с агрегацией
    @Query(value = """
        SELECT u.name, COUNT(o.id) as order_count, AVG(o.total_amount) as avg_amount 
        FROM users u 
        LEFT JOIN orders o ON u.id = o.user_id 
        GROUP BY u.id, u.name 
        HAVING COUNT(o.id) > :minOrders
        """, nativeQuery = true)
    List<Object[]> findUsersOrderStats(@Param("minOrders") long minOrders);
}
```
**Параметры запроса:**
``` java
// Позиционные параметры
@Query("SELECT u FROM User u WHERE u.name = ?1 AND u.age > ?2")
List<User> findByNameAndAge(String name, int age);

// Именованные параметры (рекомендуется)
@Query("SELECT u FROM User u WHERE u.name = :name AND u.age > :age")
List<User> findByNameAndAge(@Param("name") String name, @Param("age") int age);

// SpEL выражения (Spring Expression Language)
@Query("SELECT u FROM User u WHERE u.name = :#{#user.name} AND u.age > :#{#user.age}")
List<User> findByUserProperties(@Param("user") User user);
```
## 3. N+1 Problem (Проблема N+1)
**Теоретический материал:**
**Проблема N+1** возникает когда для загрузки основной сущности и связанных сущностей выполняется 1 запрос для основной сущности и N дополнительных запросов для каждой связанной сущности.

**Пример проблемы:**
``` java
@Entity
public class User {
    @Id private Long id;
    private String name;
    
    // LAZY загрузка по умолчанию для @OneToMany
    @OneToMany(mappedBy = "user")
    private List<Order> orders;
}

// ПРОБЛЕМА: при таком коде выполнится N+1 запросов
List<User> users = userRepository.findAll();
for (User user : users) {
    // Для каждого пользователя выполняется отдельный запрос за заказами!
    System.out.println(user.getOrders().size()); // Триггер LAZY загрузки
}
```
### **Решения проблемы N+1:**
**1. JOIN FETCH в JPQL:**
``` java
@Query("SELECT u FROM User u JOIN FETCH u.orders")
List<User> findAllWithOrders();

// Для пагинации с JOIN FETCH нужен отдельный countQuery
@Query(value = "SELECT u FROM User u JOIN FETCH u.orders",
       countQuery = "SELECT COUNT(u) FROM User u")
Page<User> findAllWithOrders(Pageable pageable);
```
**2. @EntityGraph:**
``` java
// Использование именованного EntityGraph
@Entity
@NamedEntityGraph(name = "User.withOrders", 
                  attributeNodes = @NamedAttributeNode("orders"))
public class User { ... }

@Repository
public interface UserRepository extends JpaRepository<User, Long> {
    @EntityGraph(value = "User.withOrders", type = EntityGraphType.FETCH)
    List<User> findAll();
    
    // Или inline определение
    @EntityGraph(attributePaths = {"orders", "orders.items"})
    List<User> findByNameContaining(String name);
}
```
**3. BatchSize для LAZY загрузки:**
``` java
@Entity
public class User {
    @OneToMany(mappedBy = "user")
    @BatchSize(size = 10) // Загружает заказы пачками по 10 пользователей
    private List<Order> orders;
}
```

**4. Проекции (DTO) с явными JOIN:**
``` java
@Query("""
    SELECT new com.example.UserOrderDTO(u.id, u.name, o.id, o.totalAmount) 
    FROM User u JOIN u.orders o
    WHERE u.active = true
    """)
List<UserOrderDTO> findActiveUsersWithOrders();
```
## 4. Кэширование (Caching)
**Уровни кэширования:**
**1. Кэширование на уровне БД:**
    - Встроенное в СУБД: Query cache, Buffer pool

    - Настройка индексов для ускорения поиска

    - Материализованные представления для сложных запросов

**2. Кэширование на уровне приложения (Spring Cache):**
**Настройка кэширования:**

``` java
@Configuration
@EnableCaching
public class CacheConfig {
    
    @Bean
    public CacheManager cacheManager() {
        return new ConcurrentMapCacheManager("users", "orders");
    }
    
    // Для Redis (рекомендуется для production)
    @Bean
    public RedisCacheManager redisCacheManager(RedisConnectionFactory factory) {
        RedisCacheConfiguration config = RedisCacheConfiguration.defaultCacheConfig()
            .entryTtl(Duration.ofMinutes(30))
            .disableCachingNullValues();
        
        return RedisCacheManager.builder(factory)
            .cacheDefaults(config)
            .build();
    }
}
```
**Использование @Cacheable:**
``` java
@Service
public class UserService {
    
    // Простое кэширование
    @Cacheable(value = "users", key = "#id")
    public User findById(Long id) {
        return userRepository.findById(id).orElseThrow();
    }
    
    // Условное кэширование
    @Cacheable(value = "users", key = "#id", unless = "#result == null")
    public User findByIdConditional(Long id) {
        return userRepository.findById(id).orElse(null);
    }
    
    // Кэширование с условием
    @Cacheable(value = "users", key = "#id", condition = "#id > 10")
    public User findByIdIfGreaterThan10(Long id) {
        return userRepository.findById(id).orElseThrow();
    }
    
    // Кэширование списка (сложный ключ)
    @Cacheable(value = "users", key = "T(java.util.Arrays).toString(#ids)")
    public List<User> findByIds(List<Long> ids) {
        return userRepository.findByIdIn(ids);
    }
}

// Обновление и удаление из кэша
@Service
public class UserService {
    
    @CacheEvict(value = "users", key = "#user.id")
    public User update(User user) {
        return userRepository.save(user);
    }
    
    @CacheEvict(value = "users", allEntries = true) // очистить весь кэш
    public void clearCache() {}
    
    @Caching(evict = {
        @CacheEvict(value = "users", key = "#id"),
        @CacheEvict(value = "users-list", allEntries = true)
    })
    public void delete(Long id) {
        userRepository.deleteById(id);
    }
}
```
**3. Кэширование на уровне браузера (HTTP caching):**
``` java
@RestController
public class UserController {
    
    @GetMapping("/users/{id}")
    public ResponseEntity<User> getUser(@PathVariable Long id) {
        User user = userService.findById(id);
        
        return ResponseEntity.ok()
            .cacheControl(CacheControl.maxAge(30, TimeUnit.MINUTES))
            .eTag(String.valueOf(user.getVersion())) // Для conditional requests
            .body(user);
    }
}
```
## 5. Пагинация
### **1. LIMIT/OFFSET пагинация (традиционная):**
**Преимущества:**

- Простая реализация

- Возможность прыгать на любую страницу

**Недостатки:**

- Медленная на больших OFFSET (БД должна пропустить N записей)

- Проблемы с консистентностью при изменении данных

**Реализация в Spring:**
``` java
@Repository
public interface UserRepository extends JpaRepository<User, Long> {
    Page<User> findByActiveTrue(Pageable pageable);
    
    // Кастомный запрос с пагинацией
    @Query("SELECT u FROM User u WHERE u.age > :minAge")
    Page<User> findUsersOlderThan(@Param("minAge") int minAge, Pageable pageable);
}

@Service
public class UserService {
    public Page<User> getUsers(int page, int size, String sortBy) {
        Pageable pageable = PageRequest.of(page, size, Sort.by(sortBy));
        return userRepository.findByActiveTrue(pageable);
    }
}

@RestController
public class UserController {
    @GetMapping("/users")
    public Page<User> getUsers(
            @RequestParam(defaultValue = "0") int page,
            @RequestParam(defaultValue = "20") int size,
            @RequestParam(defaultValue = "name") String sort) {
        
        return userService.getUsers(page, size, sort);
    }
}
```
### **2. Cursor-based (Keyset) пагинация:**
**Преимущества:**

- Постоянная скорость независимо от номера страницы

- Консистентность данных (устойчива к изменениям во время пагинации)

**Недостатки:**

- Невозможность прыгать на произвольную страницу

- Более сложная реализация

**Реализация курсорной пагинации:**

``` java
@Repository
public interface UserRepository extends JpaRepository<User, Long> {
    
    // Пагинация по ID (самый простой случай)
    List<User> findByIdGreaterThan(Long cursor, Pageable pageable);
    
    // Пагинация по дате создания и ID (для уникальности курсора)
    List<User> findByCreatedDateBeforeAndIdNotOrderByCreatedDateDescIdDesc(
        LocalDateTime dateCursor, Long idCursor, Pageable pageable);
    
    // Универсальный метод для составного курсора
    @Query("""
        SELECT u FROM User u 
        WHERE (u.createdDate < :dateCursor) 
           OR (u.createdDate = :dateCursor AND u.id < :idCursor)
        ORDER BY u.createdDate DESC, u.id DESC
        """)
    List<User> findNextPage(@Param("dateCursor") LocalDateTime dateCursor, 
                           @Param("idCursor") Long idCursor, Pageable pageable);
}

@Service
public class UserService {
    public List<User> getUsersCursor(LocalDateTime dateCursor, Long idCursor, int limit) {
        Pageable pageable = PageRequest.of(0, limit);
        
        if (dateCursor == null || idCursor == null) {
            // Первая страница
            return userRepository.findAllByOrderByCreatedDateDescIdDesc(pageable);
        } else {
            // Последующие страницы
            return userRepository.findNextPage(dateCursor, idCursor, pageable);
        }
    }
}

// DTO для ответа с курсором
public class CursorPage<T> {
    private List<T> content;
    private String nextCursor; // base64 encoded cursor
    private boolean hasNext;
    
    // constructor, getters
}
```

**Сравнение производительности:**
``` sql
-- LIMIT/OFFSET (медленно на больших offset)
SELECT * FROM users ORDER BY id LIMIT 20 OFFSET 10000; -- БД пропускает 10000 записей

-- Cursor-based (быстро всегда)
SELECT * FROM users WHERE id > 10000 ORDER BY id LIMIT 20; -- БД использует индекс
```
### **Рекомендации по выбору пагинации:**
- **Используйте LIMIT/OFFSET:** когда нужен произвольный доступ к страницам и набор данных небольшой

- **Используйте Cursor-based:** для бесконечной прокрутки, больших наборов данных, где важна производительность

- **Размер страницы:** 20-100 записей для оптимального баланса между производительностью и удобством
